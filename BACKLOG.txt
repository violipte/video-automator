========================== ANALISE DE VIABILIDADE ==============================

--- MULTI-SERVIDOR / ARQUITETURA DISTRIBUIDA ---

Pergunta: faz sentido dividir em mais de um servidor pra ficar mais rapido?

SITUACAO ATUAL (1 PC, 1 processo Python):
- Interface HTML inline carrega em <1s
- JSONs com light mode: ~30KB (de 400KB+)
- Gargalo REAL: FFmpeg/OpenCV (CPU/GPU 100% por 15-30min) + APIs externas
- A interface NAO e o gargalo — o render e

ONDE DIVIDIR AJUDARIA:
- Render em processo/maquina separado -> interface responsiva durante render
- LLM paralelo (multiplas pipelines simultaneas) -> roteiros mais rapidos
- Worker queue (Celery/RQ) -> desacopla tarefas pesadas do servidor web

ONDE NAO AJUDARIA:
- Interface ja e leve (sem framework frontend, sem build)
- TTS limitado pela API externa (ai33.pro), nao pelo servidor
- Dados sao JSONs pequenos, nao precisa de banco dedicado

CUSTO DA COMPLEXIDADE:
- Sincronizar estado entre processos (Redis/banco necessario)
- Manter 2+ servidores (deploy, monitoramento, restart)
- Rede entre servidores adiciona latencia
- Debugging mais dificil (logs distribuidos)

VEREDICTO:
- Para 12 videos/dia em 1 PC: NAO COMPENSA. O ganho e minimo vs complexidade.
- Para 30+ videos/dia ou multiplas maquinas: CONSIDERAR separar:
  1. Servidor web (interface + API) — leve, sempre responsivo
  2. Worker de render (FFmpeg + Whisper) — maquina com GPU dedicada
  3. Worker de LLM (pipelines) — pode rodar em paralelo
  Comunicacao via fila de tarefas (Redis + Celery ou similar)
- Para escala massiva (100+ videos): render farm com multiplas GPUs

ALTERNATIVA SIMPLES (sem multi-servidor):
- Rodar render em processo separado (multiprocessing) no mesmo PC
- Ja implementado parcialmente: orchestrator roda em thread separada
- Ken Burns ja paralelo com ThreadPoolExecutor (6 workers)
- Principal otimizacao restante: paralelizar LLM calls entre canais

Prioridade: BAIXA (reavaliar quando escalar para 30+ videos/dia)
